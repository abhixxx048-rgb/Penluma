**This document is about the small spelling mistakes that developers make every day — in commit messages, pull request descriptions, Slack messages, and code comments. It matters because a misspelled word in a professional message makes careful work look careless, and most of these mistakes come from speed and habit, not from not knowing the word.**

**The main parts explained simply:**

- **The "always misspelled" short list** — A set of about 12 words that almost every developer gets wrong at least sometimes: words like "separate", "definitely", "recommend", "occurred", and "environment". The document gives a memory trick for each one so you can fix the habit once and for all.

- **The i-before-e rule** — A simple spelling pattern: write "i" before "e" (believe, achieve, field), but flip it to "e" before "i" after the letter "c" (receive, deceive). Covers the words you type most.

- **The doubling rule** — When you add "-ing" or "-ed" to a short word where the stress lands on the last sound, you double the last letter (run → running, commit → committed). When the stress does NOT land on the last sound, you do not double it (open → opening). One quick pattern to check.

- **Words that sound alike but mean different things** — Pairs that trip people up: "which" vs "witch", "parameter" (a function input) vs "perimeter" (a boundary), and "a lot" (two words, always) vs the non-word "alot". Swapping these in a PR looks like a knowledge gap, not just a typo.

- **Texting habits leaking into work writing** — Shortcuts like "u", "ur", "plz", and "thx" are fine in personal chats but read as unprofessional in team Slack, PR descriptions, and tickets. The fix is just writing the full word.

- **Where these mistakes actually appear** — Commit messages, PR titles, Slack standups, code comments, and bug reports. The document shows real before-and-after examples for each place so the habit change feels concrete.

- **Practice drills with answers** — Twelve short exercises that reproduce the exact mistakes from the "See it / fix it" table. You can use these to check whether a correction has actually stuck.

**What to do with this:** Pin the "always misspelled" word list somewhere visible (a sticky note or phone note). Turn on spell-check in your code editor and in whatever tool you use to write PR descriptions. Then before sending any Slack message to the team, spend two seconds reading it back — spell out "you", "your", "please", and "thanks" in full.
