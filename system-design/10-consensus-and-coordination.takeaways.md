**This document explains how a group of computers can agree on one correct answer — even when some of them crash, slow down, or lose messages. It matters because almost every feature that "must never go wrong" (picking one leader, locking a shared record, keeping replicas in sync) depends on this agreement machinery working reliably.**

**The main parts explained simply:**

- **Why agreement is hard** — Computers can't tell if a peer is dead or just very slow. This single fact makes perfect, always-on agreement mathematically impossible, and everything else in this document is a practical workaround for that limitation.
- **FLP impossibility** — A 1985 proof that says: you cannot guarantee both "we will always decide" and "we will always be safe" when even one machine might crash. Real systems keep the "always safe" side and accept that progress may pause briefly during failures.
- **Paxos** — The original agreement algorithm. It works by having machines vote in rounds until a majority agrees. Proven correct but famously hard to turn into real code because many details are left undefined.
- **Raft** — A cleaner, fully specified version of the same idea. One machine is "leader" and tells the others what to record; if it dies, the others elect a new leader within a second or two. This is what etcd, Consul, and CockroachDB actually run.
- **ZooKeeper / etcd / Consul** — Ready-made services that run Raft or a similar protocol for you. Your app asks them to store small, critical facts (who is the leader? is this lock held?) and they guarantee those facts are consistent across machines. Think of them as a tiny, ultra-reliable shared notebook.
- **Leader election** — A recipe built on top of the above: whoever grabs a special key first is the leader; if they go quiet, the key expires and someone else takes over automatically.
- **Distributed locks — and why they fail** — A "lock" across machines is not safe by itself. A machine can pause (garbage collection, network hiccup), the lock expires, a second machine grabs it, and now two machines both think they own it. The only reliable fix is a fencing token — a number that rises with each new grant; the shared resource rejects any write carrying a stale, lower number.
- **Leases** — A lock with a built-in expiry time. The holder must keep renewing it; if it crashes, the lease simply runs out and someone else takes over. Leaders can use a valid lease to answer reads instantly without asking the whole group first — a useful speed-up.
- **Failure detection (Gossip / SWIM / Phi-accrual)** — Methods for spreading "who is alive" information across hundreds or thousands of machines cheaply. Gossip works like rumour — each machine tells a few neighbours, and the news spreads quickly. Phi-accrual gives a sliding suspicion score instead of a hard timeout, so the system adapts to slow networks without false alarms.
- **Clocks for ordering** — Wall clocks on different machines drift and can jump, so you cannot use them to decide which event happened first. Lamport timestamps (a simple counter), vector clocks (one counter per machine), and Hybrid Logical Clocks (a mix of real time + counter) each give stronger guarantees at higher cost. Google's Spanner uses GPS-backed atomic clocks and waits out the remaining tiny uncertainty before confirming a transaction.
- **Common traps** — Using Redis as a correctness lock (it is not one), running an even number of consensus nodes, storing large data in etcd/ZooKeeper, and hand-coding Raft while missing a subtle safety rule that causes silent data overwrite.
- **Kubernetes case study** — Kubernetes stores all cluster state in a 3-or-5-node etcd cluster. Every write goes through Raft, so the state is always consistent. Controllers use etcd leases for their own leader election, and every object carries a version number that rejects stale writes — the same fencing-token idea, built into the API.

**What to do with this:** When a system must never do something twice or must have exactly one active worker, reach for a proper consensus-backed service (etcd or ZooKeeper) rather than a plain Redis lock. Always pair it with a fencing token — let the resource reject stale writers — because a lock alone cannot stop a paused process from waking up and causing damage.
