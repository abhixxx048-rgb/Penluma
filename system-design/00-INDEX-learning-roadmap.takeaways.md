**This is a 20-lesson study plan for software developers who want to get very good at designing large, reliable computer systems — the kind that handle millions of users without breaking or losing data. It goes step by step from basic ideas all the way to the hardest problems that companies like Amazon and Google have to solve every day. If your developer is working through this, it will sharpen how they think when the platform needs to grow or when something goes wrong at 3 a.m.**

**The main parts explained simply:**

- **Tier 1 — The Basics (Modules 1, 2, 3, 6)** — Start here. Learn how to estimate how big a system needs to be before you build it, how the internet actually moves data between computers, how apps talk to each other, and how a "cache" works — storing copies of popular data nearby so the system doesn't keep fetching the same thing over and over.

- **Tier 2 — Databases and Data (Modules 4, 5, 7, 8)** — Learn how a database actually finds and stores your data under the hood, when to use different types of storage (not all data belongs in the same kind of database), and how to spread work across many servers so the system stays fast as it grows.

- **Tier 3 — Distributed Systems Core (Modules 9–15)** — The hardest part. When your data lives on many computers at once, keeping it correct and consistent is very tricky. These lessons explain the honest trade-offs: a system usually cannot be both always available AND always perfectly up-to-date at the same time. You have to pick.

- **Tier 4 — Running Systems Reliably (Modules 16–19)** — How to stop a system from being overwhelmed by too many requests, how to notice problems before customers do, and how to decide when to break a big app into smaller separate pieces — or when that would make things worse.

- **Tier 5 — Interview Prep (Module 20)** — A 7-step thinking method for designing any system, plus 10 worked examples such as how to build a chat app, a URL shortener, or a payment system.

- **12 Core Mental Models** — Big ideas that come up in every topic, such as "every choice is a trade-off," "failure is normal at scale, not rare — design for it," and "don't over-engineer: a single well-run database handles more than most teams ever need."

**What to do with this:**

Read the modules in order, starting with Module 01. Do the practice questions at the end of each section before moving on — if you cannot explain the idea in plain words, you have not learned it yet. Budget roughly one focused evening per module for the early tiers, and two evenings for the harder distributed-systems modules.
