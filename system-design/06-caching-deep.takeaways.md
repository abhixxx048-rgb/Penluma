**This document is about caching — a technique where your software saves a copy of information in a very fast place so it does not have to look it up from the slow database every single time. When done well, caching makes websites feel instant and protects the database from being overwhelmed. When done badly, it can secretly cause bigger outages than having no cache at all.**

**The main parts explained simply:**

- **Why caching works** — Most websites have a small number of pages or products that almost everyone looks at, and a large number that almost nobody touches. Caching that popular 5–20% of information means you can serve 90%+ of visitors without ever touching the database. The math is striking: improving your cache hit rate from 95% to 99% cuts database work by five times.

- **The layers of caching** — Caching is not one single box. It is a stack: the visitor's browser saves some things, a global content delivery network (CDN) like Cloudflare saves more, your app server holds a tiny local copy, a shared fast store like Redis holds the main cache, and even the database quietly keeps hot data in its own memory. Each layer is faster and cheaper than the one below it.

- **How you read and write with a cache (the patterns)** — The most common approach ("cache-aside") checks the cache first; on a miss it loads from the database and stores the result. On a write, it deletes the cached copy rather than updating it — this prevents a tricky bug where two writes race and leave stale data behind. Other patterns (write-through, write-behind) trade off write speed against the risk of losing data if the cache crashes.

- **Eviction policies — what gets removed when the cache is full** — The cache has limited space. LRU ("least recently used") removes whatever was not touched for the longest time and is the most common choice. LFU ("least frequently used") keeps popular items regardless of when they were last read. Modern systems use a smarter mix (W-TinyLFU) that handles both. Redis uses a fast approximation, not a perfect list, to keep overhead low.

- **Cache invalidation — keeping the copy fresh** — This is notoriously hard. The simplest approach is giving every cached item an expiry time (TTL) so it dies automatically. More precise approaches delete the cached copy the moment the real data changes. For pages assembled from many pieces (a category page containing dozens of products), you need tag-based or generation-key invalidation so one product change wipes all the pages that show it.

- **The four failure modes that turn a cache into a liability** — (1) Stampede: one popular item expires and thousands of visitors all hit the database at the same moment. Fix: let only one request reload it while the rest wait. (2) Hot key: one item is so popular it overloads the single server holding it. Fix: keep a short-lived copy on each app server so the shared cache barely sees it. (3) Penetration: attackers flood you with requests for items that do not exist, bypassing the cache every time. Fix: use a Bloom filter — a tiny memory structure that answers "definitely not in the database" in one step. (4) Avalanche: all your cached items expire at the same second (e.g. after a full restart), and the database collapses under the flood. Fix: always add a small random amount to every expiry time so they spread out.

- **Consistent hashing — splitting the cache across many servers** — When one cache server is not enough, you spread keys across many. A naive split (key number divided by server count) means adding or removing one server re-shuffles almost everything and empties the cache instantly. Consistent hashing puts servers and keys on a ring so adding one server only moves a small fraction of keys.

- **Redis vs Memcached** — Both are fast in-memory stores. Memcached is simpler and uses multiple CPU cores well — good for pure speed. Redis does far more: leaderboards, rate limiters, queues, atomic counters, and it can save data to disk so a restart does not lose everything. Most teams use Redis by default.

- **Facebook's real-world example** — Facebook used plain cache-aside with delete-on-write at billions of requests per second. They added "leases" (only one server reloads a missed item, others wait) to stop stampedes, tailed the database commit log to spread cache deletions across regions automatically, and accepted that data could be a few seconds out of date — a deliberate trade-off to keep the system fast.

**What to do with this:** Always add a small random amount to every cache expiry time (jitter) — it is a one-line change that prevents mass simultaneous expiry. And when something is not in the database, cache that "not found" answer too, or use a Bloom filter — otherwise attackers or bad requests can flood your database by bypassing the cache entirely.
